┌──────────────────────────────────────────────────────┐

## 2. 关键技术选型

### 2.1 VLM（视觉语言模型）

| 模型 | 延迟 | 成本 (输入/输出 per MTok) | 视觉理解 | 结论 |
|---|---|---|---|---|
| **Claude Sonnet 4** | ~2s | $3/$15 | ★★★★★ | ✅ 主力模型 |
| GPT-4o | ~1.5s | $2.5/$10 | ★★★★☆ | 备选 |
| Claude Haiku | ~0.8s | $0.8/$4 | ★★★☆☆ | 快速采样降级 |
| Gemini 2.5 Flash | ~1s | $0.15/$0.6 | ★★★☆☆ | 低成本高频采样 |
| 本地 Llama 3.2 Vision + Ollama | ~3-5s | 免费 | ★★☆☆☆ | 隐私模式 |

**决策**：主力 Claude Sonnet 4，可选降级到 Gemini 2.5 Flash 做高频采样（成本低 50 倍），离线场景用本地模型保底。

### 2.2 采样策略（成本控制关键）

| 模式 | 频率 | 分辨率 | 月成本估算（每天 8h） |
|---|---|---|---|
| 活跃工作 | 每 10s | 截屏 512px + 摄像头 256px | ~$18/月 |
| 空闲/离开 | 每 30s | 截屏 512px + 摄像头 256px | ~$6/月 |
| 离线/隐私 | 本地 Llama | — | $0 |

**决策**：按活跃度动态调整采样频率，月度 API 成本控制在 $20 以内。

### 2.3 隐私方案

```
摄像头原始帧 → 本地 MediaPipe Face Detection（仅检测人脸位置）
    → 对人脸区域做高斯模糊
    → 模糊后的图片送 VLM
    → VLM 看到模糊的人脸但能判断「有人在、表情大致如何」
    → 原始帧不离开本地
```

截屏方面：自动识别并裁剪/模糊聊天窗口、密码管理器、银行页面等敏感区域后再发送。

### 2.4 桌面端框架

| 方案 | 优势 | 结论 |
|---|---|---|
| Python + PyQt | 纯 Python，原生性能，系统托盘 + 快捷键 | ✅ |
| Electron | 跨平台、UI 丰富 | 过重 |
| Tauri | 轻量 | 学习成本高 |

**决策**：Python 全栈（PyQt system tray + pystray），核心逻辑纯 Python。

## 3. 核心技术挑战 & 应对

| 风险 | 等级 | 应对方案 |
|---|---|---|
| **VLM 推理延迟** | 🟡中 | 截图降采样 512px；streaming 输出先展示首 token；预建立 HTTP/2 长连接 |
| **API 成本失控** | 🟡中 | 动态采样频率 + Gemini Flash 降级 + 本地缓存（相同场景不重复调 VLM） |
| **隐私合规** | 🔴高 | 本地人脸模糊 + 敏感窗口自动裁剪；摄像头数据绝不上传原始帧；透明 LED 指示灯；首次启动隐私引导 |
| **VLM 幻觉/误判** | 🟡中 | 决策层置信度过滤；用户反馈闭环：「这个建议不对」→ 冷却同类建议；建议措辞留有余地 |
| **暗光环境摄像头失效** | 🟢低 | VLM 自动降级：如果摄像头帧几乎全黑，仅用屏幕截图推理，场景字段标记为「光线不足」 |
| **VLM 对场景理解不准确** | 🟡中 | System prompt 精调 + few-shot 示例；PoC 阶段收集 100 个真实场景做准确率测试 |

## 4. PoC 验证计划（VLM 方案）

```
Week 1: VLM 基础链路
  □ 截屏 + 摄像头拍照 → Claude API → 解析 JSON 输出
  □ 验证 VLM 能否准确描述「用户在写代码/吃饭/看视频/离开」
  □ 测试 50 个场景，准确率目标 > 80%

Week 2: 决策层 + UI
  □ 决策层频率控制 + 打扰等级映射
  □ 系统托盘 + Toast 弹窗原型
  □ 定时器：久坐/休闲/日程轮询

Week 3: 核心场景端到端
  □ 编程纠错：VLM 能否识别语法错误和缺少校验
  □ 就餐检测：VLM 能否识别吃饭并触发视频
  □ 久坐提醒：计时器 + VLM 确认在座 → 弹窗
  □ 会议提醒：日历接入 + 弹窗

Week 4: 内部试用 + 决策
  □ 自己连续用 3 天，记录误判率和打扰率
  □ 延迟 < 3s，场景判断准确率 > 80%，误打扰率 < 30%
  □ → GO / NO-GO
```
