# DeskMate 迭代文档

> 本文档作为 DeskMate 后续产品、技术、工程迭代的统一记录入口。之后每次迭代建议都在本文档中追加或更新，避免 PRD / Roadmap / 技术方案分散失控。

## 0. 当前项目状态快照

### 当前阶段

项目当前处于 **产品方案与技术可行性设计阶段**，尚未进入可运行工程阶段。

### 当前已有内容

- `docs/BRD.md`：商业需求文档。
- `docs/PRD.md`：产品需求文档。
- `docs/Tech_Feasibility.md`：技术可行性报告。
- `docs/Project_Roadmap.md`：项目路线图。
- `main.py`：目前仍是 PyCharm 默认示例脚本，暂无实际业务逻辑。
- `11.py`：暂无有效实现。

### 当前核心判断

DeskMate 的长期方向有潜力：

> 一个能理解用户桌面工作与生活状态的低打扰 AI 伴侣。

但当前最大问题是：

1. MVP 范围偏大。
2. 工程实现尚未开始。
3. 对 VLM 端到端决策依赖偏高。
4. 隐私、成本、打扰体验风险较高。
5. 摄像头、情绪识别、自动娱乐服务等能力不适合作为最早期验证重点。

## 1. 产品定位收敛

### 原始定位

> 一个「看着你工作与生活」的桌面 AI 伴侣，通过摄像头 + 屏幕截图，用 VLM 端到端理解你当前在做什么、情绪如何，然后主动给出恰当建议。

### 建议收敛后的早期定位

> 面向独立开发者的低打扰桌面 AI 观察员，基于屏幕上下文理解当前工作状态，在合适时给出轻量建议。

### 早期不建议强调的定位

暂时不要主打：

- 摄像头常驻观察。
- 情绪识别。
- 自动打开视频网站。
- 全天候生活管家。
- 工作效率审计。
- 团队状态监控。

这些方向可以作为长期扩展，但不适合作为第一个验证闭环。

## 2. 核心风险清单

### R1. MVP 范围过大

当前文档中的 MVP 同时包含：

- 截屏。
- 摄像头。
- VLM。
- 决策层。
- 系统托盘。
- Toast 通知。
- 久坐提醒。
- 日程提醒。
- 就餐检测。
- 娱乐服务。

风险：4 周内实现难度高，且每个模块都会引入独立不确定性。

建议：第一阶段只做 **屏幕截图 + VLM 分析 + 建议展示 + 用户反馈记录**。

---

### R2. VLM 端到端决策不稳定

原方案倾向于让 VLM 同时判断：

- 用户在做什么。
- 是否应该提醒。
- 应该提醒什么。
- 提醒优先级。

风险：

- 同一画面多次输出可能不一致。
- 建议质量不稳定。
- 容易产生误打扰。
- 本地系统缺乏可解释控制。

建议改为：

```text
VLM 负责观察和解释。
本地策略负责是否打扰、何时打扰、如何打扰。
```

---

### R3. 隐私风险高

项目涉及：

- 屏幕截图。
- 摄像头画面。
- 代码内容。
- 聊天窗口。
- 日历事件。
- 用户工作习惯。
- 潜在情绪状态。

建议原则：

1. 默认不启用摄像头。
2. 默认不保存原始截图。
3. 明确展示即将发送给模型的内容。
4. 支持敏感应用黑名单。
5. 支持一键暂停采集。
6. 所有日志默认脱敏。

---

### R4. 成本估算需重新验证

当前文档中设想每 10 秒调用一次 VLM。

按每天 8 小时计算：

```text
8 小时 × 60 分钟 × 6 次/分钟 = 2880 次/天
```

每月 20 个工作日：

```text
2880 × 20 = 57600 次/月/用户
```

风险：如果每次都上传图片给高质量 VLM，成本和延迟都可能不可控。

建议：

- 从手动触发开始。
- 再做低频自动采样。
- 使用本地事件触发，而不是固定轮询。
- 前台窗口、键鼠停顿、屏幕 hash 变化可作为是否调用 VLM 的前置判断。

---

### R5. 情绪识别建议后置

情绪识别容易误判，且涉及较高隐私和伦理敏感度。

建议将「情绪日记」降级为：

- 工作负荷记录。
- 会议密度提醒。
- 休息节律建议。
- 用户自评状态。

避免早期使用类似：

> 我检测到你连续三天情绪低落。

更推荐：

> 今天会议连续 3 小时，要不要休息一下？

## Iteration 1：手动截屏 + VLM 分析 PoC

### 目标

验证 VLM 是否能从当前屏幕截图中理解用户工作状态并给出有用建议。

### 范围

只做屏幕截图，不做摄像头。

### 用户流程

```text
用户手动执行命令
  ↓
程序截取当前屏幕
  ↓
本地进行基础脱敏 / 降采样
  ↓
发送给 VLM
  ↓
解析结构化 JSON
  ↓
在终端展示分析结果和建议
```

### 建议命令

```bash
python -m deskmate analyze-screen
```

### 目标输出格式

```json
{
  "scene": "coding",
  "confidence": 0.82,
  "observations": [
    "用户正在编辑 Python 项目",
    "屏幕中打开了代码编辑器"
  ],
  "possible_issue": "当前项目入口仍较简单，尚未形成实际业务结构",
  "should_suggest": true,
  "suggestion": "建议先建立 src/、requirements.txt、README 和基础测试目录。",
  "priority": 2
}
```

### 验收标准

- 能成功截屏。
- 能调用 VLM。
- 能得到结构化结果。
- VLM 返回非 JSON 时不会崩溃。
- 支持保存本次分析日志。
- 默认不保存原始截图，除非用户显式开启 debug。

---

## Iteration 2：结构化解析与容错

### 目标

让 VLM 输出变得可控、稳定、可测试。

### 重点能力

- JSON schema 校验。
- 字段缺失兜底。
- priority 范围修正。
- confidence 范围修正。
- 非 JSON 输出修复。
- 错误日志记录。

### 测试重点

- VLM 输出空字符串。
- VLM 输出 Markdown 包裹 JSON。
- VLM 输出字段缺失。
- VLM 输出非法 priority。
- VLM 输出中文解释 + JSON 混合内容。

### 验收标准

- Parser 单元测试覆盖主要异常情况。
- 程序不会因为模型格式异常直接崩溃。
- 无法解析时返回明确错误状态。

---

## Iteration 3：本地决策层与低打扰策略

### 目标

把「是否提醒」从 VLM 中部分剥离，交给本地策略层控制。

### 核心策略

- 同类建议 30 分钟内不重复。
- priority 1-2：只记录。
- priority 3：轻提示。
- priority 4-5：弹窗。
- 用户忽略后，同类建议冷却 2 小时。
- 用户关闭某类建议后，默认不再主动提示。

### 建议数据结构

```json
{
  "suggestion_type": "coding_structure",
  "priority": 2,
  "cooldown_key": "coding_structure:init_project",
  "should_notify": false,
  "reason": "priority 低，仅记录到日志"
}
```

### 验收标准

- 有 `RateLimiter`。
- 有 `DecisionPolicy`。
- 有对应单元测试。
- 冷却策略可配置。

---

## Iteration 4：用户反馈闭环

### 目标

开始收集「建议是否有用」，为后续优化提示策略和产品方向提供依据。

### 支持反馈

- 有用。
- 没用。
- 稍后提醒。
- 不再提醒此类。

### 事件日志建议

```json
{
  "timestamp": "2026-06-01T12:00:00",
  "event_type": "suggestion_feedback",
  "scene": "coding",
  "suggestion_type": "coding_structure",
  "priority": 2,
  "user_action": "useful"
}
```

### 验收标准

- 每条建议可以被反馈。
- 反馈会写入本地 SQLite 或 JSONL。
- 后续决策层可以读取反馈结果调整冷却。

---

## Iteration 5：轻量桌面 UI

### 目标

从 CLI 过渡到桌面常驻应用。

### 范围

- 系统托盘。
- 手动触发「分析当前屏幕」。
- 最近建议历史。
- 暂停采集按钮。
- 隐私设置入口。

### 暂不做

- 自动高频采样。
- 摄像头。
- 情绪日记。
- 自动打开网站。

### 验收标准

- 托盘应用可以启动和退出。
- 用户可以手动触发分析。
- 用户可以暂停全部采集。
- 用户可以查看最近 N 条建议。

---

## Iteration 6：低频自动采样

### 目标

在不打扰用户的前提下，从手动分析过渡到半自动建议。

### 触发条件建议

优先采用事件触发，而不是固定每 10 秒调用：

- 前台窗口变化。
- 用户停顿超过 30 秒。
- 打开 IDE 或浏览器开发相关页面。
- 屏幕 hash 发生明显变化。
- 用户主动开启「专注辅助模式」。

### 初始采样策略

```text
默认：不自动采样。
开启后：最低 60 秒间隔。
同类场景：30 分钟内不重复弹窗。
敏感应用：永不采样。
```

### 验收标准

- 用户可开关自动采样。
- 自动采样有最小间隔。
- 敏感应用不会触发截图上传。
- 有成本统计记录。

---

## Iteration 7：久坐与日程提醒

### 目标

加入不强依赖 VLM 的高确定性功能。

### 久坐提醒

- 基于键鼠活动和前台状态计时。
- 超过 45 分钟提醒。
- 用户离开或锁屏后重置。

### 日程提醒

- 先支持手动 JSON/YAML 日程。
- 后续再接 Outlook / Google Calendar。

### 验收标准

- 离线可用。
- 不需要 VLM 也能运行。
- 提醒频率可配置。

---

## Iteration 8：摄像头能力实验

### 目标

仅作为实验能力验证，不作为默认功能。

### 原则

- 默认关闭。
- 用户主动开启。
- 本地预览。
- 人脸模糊后才允许上传。
- 不保存原始帧。

### 优先验证场景

- 是否有人在座位。
- 是否离开。
- 光线是否不足。

### 暂不验证

- 情绪识别。
- 复杂肢体动作。
- 就餐自动服务。

### 验收标准

- 摄像头权限透明。
- 本地模糊可见效果可验证。
- 用户可以随时关闭。

## 4. 暂缓功能池

以下功能不建议进入早期 MVP：

| 功能 | 暂缓原因 |
|---|---|
| 情绪日记 | 隐私与伦理风险较高，误判成本高 |
| 自动打开视频网站 | 产品主线弱，用户控制感不足 |
| 时间审计周报 | 需要稳定长期数据，早期数据质量不足 |
| 代码复盘日报 | 可作为独立功能，但不是验证 VLM 场景理解的核心 |
| 离线 Llama Vision | 工程成本较高，早期不是关键瓶颈 |
| B2B 团队面板 | 与个人桌面助手 MVP 差异较大 |
| 硬件一体机 | 需要先验证软件价值 |

## 5. 工程规范建议

### 依赖管理

建议先使用 `requirements.txt`，后续项目复杂后再迁移到 `pyproject.toml`。

### 环境变量

`.env.example` 至少包含：

```text
ANTHROPIC_API_KEY=
DESKMATE_DEBUG=false
DESKMATE_SAVE_SCREENSHOTS=false
```

### 日志原则

默认记录：

- 时间。
- 场景类型。
- 建议类型。
- priority。
- 用户反馈。
- 成本估算。
- 延迟。

默认不记录：

- 原始截图。
- 原始摄像头帧。
- 完整代码内容。
- 聊天内容。
- 日历详情全文。

### 测试优先级

优先测试：

1. VLM 输出解析。
2. 决策冷却策略。
3. 隐私黑名单。
4. 事件日志写入。
5. 错误兜底。

## 6. 成功指标

### PoC 阶段指标

| 指标 | 目标 |
|---|---|
| 手动分析成功率 | > 90% |
| JSON 解析成功率 | > 95% |
| 单次分析延迟 | < 5 秒 |
| 单次调用成本 | 可记录、可估算 |
| 建议主观有用率 | > 30% |

### MVP 阶段指标

| 指标 | 目标 |
|---|---|
| 日均主动提醒 | < 10 次 |
| 用户主动关闭率 | < 20% |
| 建议采纳率 | > 30% |
| 误打扰率 | < 30% |
| 默认隐私模式可用 | 是 |
